# 0036. Real Bravura SMuFL glyphs for the key-signature chip and chord-tone staff

- **Status**: Accepted
- **Date**: 2026-06-22

## Context

Two music-notation surfaces drew **hand-rolled, simplified caricatures**
of SMuFL glyphs:

- the inline `{key}` key-signature chip — `KeySignatureGlyph`
  (`packages/react/src/music-glyphs.tsx`) and its HTML sister
  `key_signature_svg` (`crates/render-html/src/music_glyphs.rs`); and
- the chord-editor constituent-note staff — `ChordStaff`
  (`packages/react/src/chord-staff.tsx`).

The treble clef was a few Bézier segments approximating the gClef; sharps
and flats were stroked line/curve primitives; noteheads were rotated
ellipses. At chip / footer size the clef in particular read as a distorted
squiggle (#2697).

Meanwhile the iReal renderer already draws **real Bravura SMuFL outlines**
baked as inline SVG `<path>` data per
[ADR-0014](0014-bravura-glyphs-as-svg-paths.md) (segno / coda / fermata,
extracted deterministically by `scripts/extract-bravura-paths.py`). The
question this ADR settles is how to fix the distorted glyphs given the
project's constraints, and — because the maintainer asked the fix to be
chosen "thinking about the service's future" — whether a full music-
notation library should be adopted instead.

## Decision

Extend ADR-0014's path-baked-Bravura approach to the key-signature chip
and the chord-tone staff. Add the gClef (U+E050), accidentalSharp
(U+E262), accidentalFlat (U+E260), and noteheadBlack (U+E0A4) outlines as
baked SVG `<path>` data, generated by the same
`scripts/extract-bravura-paths.py` (new `--target html` / `--target react`
selectors) from the same SHA-256-pinned upstream font:

- `crates/render-html/src/bravura.rs` — gClef / sharp / flat for the HTML
  `{key}` chip.
- `packages/react/src/bravura-glyphs.ts` — gClef / sharp / flat /
  noteheadBlack for `<KeySignatureGlyph>` and `<ChordStaff>`.

Glyphs map from font units into each surface's user-space through a shared
`smuflTransform` / `smufl_transform` helper (scale by `staffSpace / 250`,
flip Y, anchor a font point onto a target point). The React
`<KeySignatureGlyph>` and the Rust `key_signature_svg` stay byte-equivalent
sister sites per `.claude/rules/renderer-parity.md`.

Do **not** adopt a JavaScript music-notation engine (VexFlow, abcjs,
OpenSheetMusicDisplay, Verovio) for these surfaces.

## Rationale

- **Renderer parity forbids a JS-only engine here.** ChordPro key-signature
  rendering ships from both `crates/render-html` (CLI / FFI / Action / VS
  Code) and `@chordsketch/react`, which
  `.claude/rules/renderer-parity.md` requires to produce equivalent
  output. A JavaScript engine has no Rust counterpart, so it would split
  the two surfaces — the exact failure class the parity rule exists to
  prevent.
- **The use case is icon-sized glyphs, not engraving.** The surfaces draw a
  ~1.1em key-signature chip and a short arpeggiated chord-tone staff — no
  rhythm, beaming, multi-measure layout, or part alignment. A full
  engraving engine is overkill for four glyph shapes.
- **ADR-0014's measurements already hold.** Each glyph is sub-2 KB of raw
  path data — smaller than a WOFF2 subset and far smaller than the ~500 KB
  Bravura binary — and inline `<path>` needs no `fontdb` registration in
  the PNG / PDF pipelines. The React package keeps its no-forced-font-
  download posture.
- **One generator, one pinned font.** Reusing
  `scripts/extract-bravura-paths.py` keeps every baked glyph reproducible
  from the same SHA-256-pinned upstream, so the three sister files cannot
  silently drift in provenance.

## Consequences

**Positive**

- The clef, accidentals, and noteheads are now true SMuFL shapes on every
  surface; the distorted-squiggle clef is gone.
- The caricature path constants and the bespoke stroked sharp/flat/notehead
  primitives are replaced by glyph data + a small transform helper.
- Accidentals in the chord-tone staff now reserve their own horizontal slot
  (real Bravura accidentals are ~1 staff space wide), so they no longer
  collide with the previous tone.

**Negative**

- The path data is opaque (long `d=` strings) — same trade-off ADR-0014
  accepted; mitigated by the regenerator script and the provenance comments.
- The project now bakes **7 distinct Bravura glyphs** across three sister
  files (was 3). ADR-0014's watch signal ("more than ~10 SMuFL glyphs
  tilts the analysis back toward bundling the font binary") is now closer;
  see References.
- OFL-1.1 attribution obligations extend to `crates/render-html` and
  `@chordsketch/react` — discharged via their `LICENSE-OFL.txt`, the
  project `NOTICE`, and `license = "MIT AND OFL-1.1"` metadata.

## Alternatives considered

### Alternative 1: Adopt VexFlow / abcjs / OpenSheetMusicDisplay

Full-featured JS notation libraries. Rejected: JavaScript-only (breaks
Rust ↔ React renderer parity), hundreds of KB, and built for full scores
rather than icon-sized chips.

### Alternative 2: Adopt Verovio (C++ → WASM, SMuFL/MEI)

The strongest candidate *if* the service grows into full sheet-music
rendering (melody lines, rhythm, multi-measure engraving). Rejected for
**now**: it is a heavy engine, still has no Rust-renderer counterpart, and
adopting it would require reopening the renderer-parity model. Recorded as
a watch signal rather than a rejection on the merits — if full-notation
rendering becomes a product goal, a successor ADR should revisit this.

### Alternative 3: Improve the hand-drawn caricatures

Keep stroked primitives but draw them more carefully. Rejected: it does not
reach true SMuFL fidelity, and ADR-0014 already established that baked
outlines cost less code than a faithful approximation while looking
correct.

### Alternative 4: Bundle the Bravura font via `@font-face` / a notation font

Rejected for the same measured reasons as ADR-0014 (per-render byte cost,
`fontdb` plumbing in the PNG/PDF backends, forced font download in the
React bundle).

### Watch signals (revisit if any of these change)

- **Full sheet-music rendering becomes a product goal** (melody lines,
  rhythm, beaming, multi-measure engraving). At that point Verovio (or an
  equivalent engine) should be re-evaluated in a successor ADR, including
  how it coexists with — or replaces — the renderer-parity model.
- **The baked-glyph count exceeds ~10** (ADR-0014's threshold). The project
  is at 7; a few more glyphs would warrant re-checking whether a pinned
  font subset is cheaper than path data.
